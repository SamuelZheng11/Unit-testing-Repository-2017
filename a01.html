Requirements

For this assignment, you are to create tests to determine whether code written by someone else is correct. You will be given a number of jar files, all of which claim to provide a class for representing a model of inheritance in Java. What a "model of inheritance in Java" means, and what the specific functionality you have to test for is, is described in the provided JavaDoc.

In fact, each implementation except one is known to have at least one fault. You have to create a set of tests (a test suite) using the JUnit framework that will detect at least one fault in all of the different implementations.

All (deliberate) faults are due to the implementation not meeting the stated requirements, that is, they are detectable by basing tests only on what you see in the JavaDoc description.

You must use JUnit 4 for your tests. Unfortunately different versions JUnit 4 provide slightly different functionality. In order to ensure your submission can be used by the automatic marking system (see below) you should use 4.12. Using an earlier version should be acceptable.

JUnit 4 provides many annotations, most of which you should avoid. The nature of this assignment is that you will not get much benefit from them, and the time spent on figuring out how to use them properly would be better spent on the assignment itself.

The one annotation you cannot avoid, @Test, can take two parameters. One of these, timeout, will possibly be useful as it limits the time a test takes (e.g. in case of infinite loops). The other, expected (exception) may not be used. I want you to demonstrate your understanding of the use of try/catch to test exceptions.

You will see in the jar files that there are more .class files than just the one for InheritanceModel.class. You do not need to test these.

Assessment

This assignment worth 5% of your overall grade for SOFTENG254. Marks will be allocated according to how well you demonstrate how well you understand how to develop good quality tests.

Primarily your understanding will be determined by how many of the implementations provided you correctly detect faults in (not, "how many faults"), and how well you explain what you are testing.

Your tests will be partially assessed by an automatic marking system. If the automatic marking system either cannot mark your submission, or marks it incorrectly due to your submission not meeting the requirements, then you will likely get zero for the assignment. (Think of it as you have been assigned to write a class that fits into another system. Your boss is unlikely to be impressed if you write a class that doesn't fit.) Note that if your class cannot be compiled, then the automatic marking system will not be able to mark it.

The assignment is marked out of 20 as follows:

  • You will get 2 marks if all your tests pass on the "correct" implementation, or if all failed tests indicate an actual fault in the "correct" implementation.
  • You will get 0.5 marks for every implementation supplied to you if your test suite indicates that that implementation has a fault. This means that one of your tests must demonstrate that the implementation does the wrong thing. It is not sufficient that JUnit indicates that the test has failed --- the test must fail and the failure is clearly due to the implementation containing a fault. (5 marks total)
  • Your tests will be run against another set of implementations that contain faults. You will get 2 marks for each implementation your test suite correctly indicates contains a fault. (10 marks total)
  • You will get 3 marks for how well you document your tests. (See "Explaining the test case" below).

In addition, you could be penalised as follows:

  1. 50% of your mark out of 20 (as described above) if your tests execute an assertX, fail, or similar method more than 200 times. This is a fairly generous limit, that is, it is much bigger than what you should need. If you feel you need more, discuss it with me. Note that it is not how many times these methods appear in your tests, but how many times they are executed (e.g if you put them in a loop).
  2. 25% of your mark out of 20 (as described above) if your test code quality is poor (See "Test Code Quality" below).
Note also that using any external means (e.g. a tool) to develop your tests is unacceptable. The goal of this assignment is not to come up with the tests, but to demonstrate that you understand how to come up with good quality tests. Use of external means will not convince me of your understanding.

Explaining the test case

It is a common situation that someone (usually you) looking at your code at some point in the future thinks "I wonder if there is a test for this weird case?" In order to help answer this kind of question, it is worth documenting your test cases. It doesn't have to be super elaborate — just a sentence or so that would help someone reading your tests figure out what general case you are trying to test. Put such documentation in JavaDoc comments for your test case methods. Documentation that makes no attempt to explain why it is worth testing what you're testing or no documentation at all are examples that are likely to lose marks. An example of poor documentation is: "Tests the case for two classes related by inheritance". This says what the test is, but not why this test is of good quality. You need to give some idea as to why you think this particular input will improve the quality of your test suite.

Test Code Quality

You should give some thought to how you organise your tests. You might even consider developing auxiliary methods to make the test methods easier to write. As well as documenting your individual methods, it is often helpful to group them according to some more general scheme to help keep track of what is being tested where. Lack of any organisation, poor test code, and code that contains debugging print statements are all examples that are likely to lose marks.
